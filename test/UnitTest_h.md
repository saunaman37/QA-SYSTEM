# 人間が実施するテスト一覧

## 対象システム
React/TypeScript + Spring Boot + MySQL 構成の Q&A システム（v1.0）

## 分類基準
以下のいずれかに該当するテストを人間実施として分離している。
- 自動テストでは判定が困難なもの
- 実機・目視での確認が適切なもの
- 実 DB 環境や実ブラウザが必要なもの

---

## H-1. 統合テスト（DB）：削除トランザクションのロールバック確認

| 項目 | 内容 |
|-----|------|
| **対象機能** | `QuestionService#deleteQuestion` |
| **確認内容** | 質問と関連回答が同一トランザクション内で論理削除されること。処理途中でエラーが発生した場合に、質問・回答ともにロールバックされ中途半端な状態にならないことを確認する |
| **手順** | 1. Docker MySQL コンテナを起動する<br>2. 質問（id=1）と関連回答（2件）を DB に登録する<br>3. 削除処理の途中で例外が発生するよう DB 側で制約を仕込むか、アプリ側でデバッグ用に例外を発生させる<br>4. `DELETE /api/questions/1` を実行する<br>5. DB を SELECT し、質問・回答ともに `deleted_at` が変更されていないことを確認する |
| **期待結果** | ロールバックにより、質問の `deleted_at` および回答の `deleted_at` がいずれも変更前の状態（null）のままであること |
| **実施タイミング** | 実 DB 環境構築後、統合テスト実施時 |
| **備考** | 単体テスト（Mockito）では同一トランザクション保証は検証できないため、ここで確認する |

## H-2. 統合テスト（DB）：タイムゾーン確認

| 項目 | 内容 |
|-----|------|
| **対象機能** | `Question` / `Answer` エンティティの `created_at` / `updated_at` |
| **確認内容** | `created_at` / `updated_at` が `Asia/Tokyo` タイムゾーンで DB に正しく保存されること |
| **手順** | 1. Docker MySQL コンテナを起動する<br>2. 質問を 1 件登録する（`POST /api/questions`）<br>3. MySQL クライアントで以下を実行する<br>　`SHOW VARIABLES LIKE 'time_zone';`<br>　`SELECT id, created_at, updated_at FROM questions ORDER BY id DESC LIMIT 1;`<br>4. 返却された `created_at` の値を現在の日本時間（JST）と照合する |
| **期待結果** | `created_at` / `updated_at` が現在の JST 時刻（±数秒以内）で保存されていること。UTC 時刻にずれていないこと |
| **実施タイミング** | Docker MySQL 起動後、初回動作確認時 |
| **備考** | `application.yml` の `hibernate.jdbc.time_zone: Asia/Tokyo` および Docker の `TZ=Asia/Tokyo` 設定が有効であることの確認 |

## H-3. UI テスト（ブラウザ）：バリデーションエラー表示確認

| 項目 | 内容 |
|-----|------|
| **対象機能** | P-02 質問登録画面、P-06 質問修正画面、回答登録モーダル（AnswerFormModal） |
| **確認内容** | バリデーションエラー発生時に、フィールド単位のエラーメッセージが画面上に表示され、どのフィールドのエラーかが利用者に伝わること |
| **手順（P-02 質問登録画面）** | 1. ブラウザで `http://localhost:5173/questions/new` を開く<br>2. 全フィールドを空のまま「登録」ボタンを押す<br>3. タイトル・質問内容・質問者名それぞれのエラーメッセージが表示されることを確認する<br>4. タイトルに 101 文字以上を入力し「登録」を押す<br>5. 文字数超過のエラーメッセージが表示されることを確認する |
| **手順（回答登録モーダル）** | 1. 質問詳細画面（`/questions/:id`）を開く<br>2. 「回答を追加」ボタンを押してモーダルを開く<br>3. 回答内容・回答者名を空のまま「登録」を押す<br>4. 各フィールドのエラーメッセージが表示されることを確認する |
| **期待結果** | 各フィールドの直下または近傍に日本語のエラーメッセージが表示される。エラーのないフィールドにはメッセージが表示されない |
| **実施タイミング** | フロントエンド実機動作確認時（バックエンドも起動済みの状態） |
| **備考** | バックエンド側のバリデーション（`@NotBlank` / `@Size`）が返す `fieldErrors` をフロントエンドが正しく画面に反映できているかの確認 |
